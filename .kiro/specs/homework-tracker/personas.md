# Personas

## Overview
ระบบนี้รองรับผู้ใช้ 4 กลุ่มที่มีสิทธิ์และเป้าหมายต่างกันชัดเจน (RBAC) ทุกคน login ด้วย Google เท่านั้น

---

## Admin (ผู้ดูแลระบบ)

**Role**: เจ้าหน้าที่/ครูประจำชั้นที่ดูแลมาสเตอร์การบ้านกลาง

**Goals**:
- ควบคุมมาสเตอร์การบ้านให้ถูกต้องและเป็นปัจจุบัน
- จัดการคำขอแก้ไขจากผู้ใช้ และมอบหมายครูเข้าชั้นเรียน
- เห็นภาพรวม progress และ audit log ของทุกครอบครัว

**Pain Points**:
- ข้อมูลการบ้านกระจัดกระจาย แก้ไขซ้ำซ้อนหลายที่
- ไม่รู้ว่าใครแก้สถานะอะไรเมื่อไหร่

**User Journey**: Login → Admin dashboard → จัดการ assignment/คำขอ/ครู → ตรวจ progress + audit

**Implications**: ต้องมีหลังบ้านครบ (CRUD, request management, audit search, teacher assignment, family overview); สิทธิ์สูงสุดแต่ไม่ผูกกับครอบครัวใดครอบครัวหนึ่ง

---

## Parent (ผู้ปกครอง)

**Role**: พ่อ/แม่/ผู้ปกครองในครอบครัว

**Goals**:
- ติดตามว่าลูกแต่ละคนทำการบ้านครบหรือยัง ขาดอะไร
- อัปเดตสถานะแทนลูกได้ และจัดการสมาชิกในครอบครัว
- แจ้ง Admin เมื่อข้อมูลมาสเตอร์ผิด

**Pain Points**:
- ตามงานลูกหลายคนพร้อมกันยาก
- ไม่อยากให้ครอบครัวอื่นเห็นข้อมูลลูกตัวเอง

**User Journey**: Login → (สร้าง/เข้าร่วมครอบครัว) → Dashboard เลือกลูก → ติ๊กสถานะ → ส่งคำขอถ้าจำเป็น

**Implications**: เห็น/แก้ได้เฉพาะครอบครัวตัวเอง (data isolation); สลับดูลูกแต่ละคน (child tabs); จัดการคำเชิญสมาชิก

---

## Child (เด็ก/นักเรียน)

**Role**: นักเรียนที่เป็นสมาชิกในครอบครัว

**Goals**:
- ดูการบ้านของตัวเองและอัปเดตสถานะเมื่อทำเสร็จ/ส่งแล้ว
- รู้ว่างานไหนใกล้ครบกำหนด/เลยกำหนด

**Pain Points**:
- ลืมว่ามีงานอะไรต้องส่งบ้าง
- UI ซับซ้อนเกินไปสำหรับเด็ก

**User Journey**: Login → Dashboard (เฉพาะของตัวเอง) → ติ๊กสถานะงานตัวเอง

**Implications**: เห็นเฉพาะการบ้าน+สถานะของตัวเอง; แก้ของพี่น้องไม่ได้; UI เรียบง่าย เป็นมิตรกับเด็ก สีบอกสถานะชัด

---

## Teacher (ครูผู้ดูแลชั้น)

**Role**: ครูที่ Admin มอบหมายให้ดูแลชั้นเรียน

**Goals**:
- ดูภาพรวมว่าเด็กในชั้นใครขาดงานอะไรบ้าง (ข้ามครอบครัว)
- แจ้ง Admin เมื่อพบข้อมูลผิด

**Pain Points**:
- ไม่เห็นภาพรวมว่าทั้งชั้นส่งงานครบหรือยัง

**User Journey**: Login → Teacher class overview (read-only) → กรองตามสถานะ → ส่งคำขอถ้าจำเป็น

**Implications**: read-only; เห็นเฉพาะชั้นที่ได้รับมอบหมาย (ไม่แบ่งเทอม); ต้อง revoke สิทธิ์ได้เมื่อเปลี่ยนครู; เห็นข้ามครอบครัวเฉพาะในชั้นนั้น

---

## Design Implications

- **Architecture**: ต้องมี RBAC ระดับ backend ทุก endpoint; การ scope ข้อมูลตาม family (Parent/Child) และตาม class assignment (Teacher); Admin ข้ามขอบเขต
- **UI/UX**: responsive (mobile + desktop); role switcher; Child UI เรียบง่ายสุด; สีสถานะสม่ำเสมอ (เทา/เหลืองอ่อน/เหลืองเข้ม/เขียว/แดง)
- **Data & Privacy**: data isolation ระหว่างครอบครัวเด็ดขาด; ลบสมาชิก/ครอบครัว → progress ถูกลบ แต่ audit log คงไว้; DB เผื่อโครงสร้าง multi-school
